Getting Started

Overview

This module's primary goal is to provide fast and smooth travel services to our customers

where they can find their rides at their own convenience.

Setting up the project

1. Install nodejs and ensure `node (>8.6 and <= 10)` and `npm` are installed
2. Run `npm install`
3. Run `npm test`
4. Run `npm start`
5. Hit the server to test health `curl localhost:8010/health` and expect a `200` response

Note:
To check for docs go to /docs while server is running(npm start) or just npm docs for generating docs.
We can specifically check for all APIs and their response by going to swagger openApi docs/api-docs.

APIs and their description -

Important: To specifically check for all APIs and their response go to swagger openApi docs/api-docs (while server is running)

The module contains APIs that serves multiple ways in adding to the smooth experience.

Note : All the APIs are in src folder inside the root folder.

/health

Description-

Use it to check the health of the server.The API returns healthy if server is up and running.

Request type: GET

Response:

200 - Server is in healthy condition.

/rides

Description-

Input info related to your ride i.e. latitude , longitude, rider-info, driver-info etc.
The API first validates the input, if valid inserts the info into the database into their respective rows
else throws the error code and message.

Request type: POST

params: start_latitude , start_longitude ,end_latitude , end_longitude ,
rider_name , ride_vehicle , traveler_name

Response:

200 - returns all the availiable rides in json placeholder.

400(Bad request) - error code and error message

/rides

Description-

The API searches for all the availible riders into the database rows and if found return them else returns error with code and message.

Request type: GET

Response:

200 - returns all the availiable rides in json placeholder.

400(Bad request) - error code and error message

/rides/:{id}

Description-

Use it to search for a specific availiable ride.

Request type: GET

params: filter for ride

Response:

200 - returns particular ride if availible in json placeholder.

404(Not found) - error code and error message

Tooling

Several tooling methods have been used in order to keep code quality consistent and error free.
These tooling include -

1. Eslint

Eslint combined with prettier ensures that a same consistency is maintained in the code and the module is bug free.

npm run lint checks for the errors/warnings in the code while npm run lint --fix fixes most of the errors that is automatically fixable.

Eslint runs itself on commiting changes on the files changed and on push runs on the whole module.

2. nyc

nyc paired with tests have been used to generate detailed coverage on the tests that is
save in coverage directory and can be view in web format.

npm test to run tests and generate coverage

3. Git hooks

Git hooks like pre-push and pre-commit have been imlemented using lint-staged and husky.

Before commiting changes Eslint runs itself on the files changed and on push runs on the whole module.

4. Winston

All logs are generated in error.logs file in logs directory using winston.

Pagination

Pagination has been added to GET rides API in app.js file in src directory along with tests(in tests directory).
